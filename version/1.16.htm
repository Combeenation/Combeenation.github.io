Version 1.16 Highlights

List of all changes

Introducing drafts

History

Prior to this release each configurator had a development version and several published versions. The latest published version was used for the production configurator.

The continually existing dev version (v0) will be published to a new version (v23)

The problem with this solution was that the the product manager had only one development configurator in which (s)he could implement new features, fix bugs, test new checkout solutions, ... .

It was impossible to work on several issues in parallel and deploy them separately. For example if someone currently worked on a new layout and an urgent bug occurred, the layout work had to be discarded or finished completely before the configurator was safe for publishing.

Drafts

This brought us to the idea that we now offer up to five drafts per configrator. A draft is an entry point for developing new features for a configurator. Multiple product managers can now work in parallel on one configurator (e.g. one fixes an urgent checkout bug, where the other introduces a new feature).

One of the two drafts can be published to a new version. New drafts can be created based on the currently published version.

Info

The changes won't be merged into each other, each drafts stands for its own and (if published) will replace the currently active configurator. Therefore changes made in one draft need to be applied manually in other drafts.

Each draft has a preview url which can be used to test integration in the parent page or checkouts. A product manager can work on a draft as long as (s)he pleases, but the lifetime of a drafts ends as soon as the configurator will be published. The publishing always marks the end of a (long running) feature/bug fix/..., therefore we are deleting the draft afterwards.

New drafts can be created if the configurator has been published at least once, because the source of a new draft is always the currently published version of the configurator.

Backend UI udpate

The new draft feature also comes with a backend UI update. Below you can see the old and the new configurator overview in comparison.

Old configurator overview

Updated configurator overview

As you can see we still have our list of configurators on the left side. In the center we now grouped up the actions, starting with the configurator based operations on the top (e.g. withdraw, the history of all versions, information about the last publishing).

The next section is the essential part of this update and displays all the drafts for the given configurator. Each draft has several actions:

  • Edit draft ... opens the draft in the configurator editor
  • View draft in editor ... opens the draft readonly in the configurator editor, useful if you just need to look up something without causing any changes
  • Preview draft ... opens the draft configurator, used for testing and integration
  • Delete draft ... unsurprisingly this will delete the draft
  • Create new configurator based on draft ... creates a new configurator based on the selected draft

At the end we show you some embed information and the URL for the currently published configurator.

Configurator deletion

You can now restore a recently deleted configurator up to seven days after the deletion took place.

This feature comes also with a Combeenation performance boost. Several actions like deleting configurators, deleting resources and so on now happen in the blink of an eye. Our system later will take care of deleting those items once and for all.
