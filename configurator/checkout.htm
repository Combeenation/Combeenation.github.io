Checkout

Combeenation supports multiple checkout types out of the box.

During the checkout you can put the users configuration into one of the popular shopping carts like Magento or WooCommerce. Alternatively you can send an E-Mail to one or more recipients with content of your choice.

If those checkout types don't fit your needs, there are also some more advanced checkout types like REST, FTP or Do it yourself checkout.

Basic checkout configuration

You can create a shop for the checkout type of your liking by clicking on the Plus icon in the upper right corner of the CheckoutComponent editor.

Whilst you can create multiple checkout shops, there's always exactly 1 active shop at a time.

The active shop is determined by the resulting value of the Hive script in the CheckoutComponent's Shop property. Since this Hive script can also take its input into consideration, there are many different ways to define which shop should be activated. E.g. by selected preset, by given URI paramter, by the wizard page on which the checkout was initiated, ...

Detailed checkout configuration

You can set the checkout details like Magento endpoints or recipient mail addresses etc. by selecting the shop in the Shops list and setting according values for the properties shown in the list below.

How to trigger the checkout

The checkout can be triggered using signals & slots in the design editor.

Usually you'll bind the CheckoutComponent's Do checkout slot to the clicked signal of a button.

Using the Do checkout slot's first parameter Selected shop, you can control which shop will be used if multiple shops exist.

The given value of the Selected shop parameter is set as input of the CheckoutComponent's Shop property. If you leave it empty the input is left untouched and remains the same as set in the CheckoutComponent editor. If not empty, Selected shop is usually the name of the shop you'd like to execute.

Warning!

The Hive rule of the CheckoutComponent's Shop property must actively use its input for the Selected shop paramter to actually take effect.

Info

When creating a new configurator, a default SigSlo already exists which performs the checkout when the user clicks the "Order now" button.

Customizing the checkout load mask

By default, the configurator shows a load mask with the text Adding configuration to cart... when the checkout is started. This load mask is never hidden by default and will stay there until the user leaves the site or closes the tab or browser.

Most checkouts redirect to the shopping cart after the checkout succeeded. This redirect also hides the load mask.

You can customize the message shown in the load mask using some signals & slots which are provided by the configurator control and the CheckoutComponent. When creating a new configurator, the following SigSlo is responsible for showing the default checkout load mask: Checkout.CheckoutStarted -> Configurator.ShowLoadMask('Adding configuration to cart...')

Confirmation after checkout

If the configurator is embedded into a shop system with a checkout cart, the configurator page is usually redirected to the cart after the checkout. This redirect is either triggered by the parent page or some custom Javascript in the configurator's parent page plugin.

For this scenario you don't have to implement any special handling. The configurator's default SigSlos show the Adding configuration to cart... load mask which is not hidden until the redirect to the cart happens.

If no redirect to a shopping cart happens (e.g. when performing a mail checkout), you'll most probably want to hide the load mask and show some sort of confirmation after the checkout succeeded (e.g. after the mail has been sent). You can do this by using the CheckoutComponent's Checkout succeeded signal and the configurator control's hide load mask slot. You can also show your own UI with some confirmation using the CheckoutComponent's Checkout succeeded signal.

Continuing configuration after checkout

If needed, you can keep the configurator open for further configuration after the checkout. A new configuration will be started automatically to protect the checked out one from any modifications. This change is completely transparent and should not be noticed by the user. When this new configuration is checked out, it replaces the previous one.

You only have to manually hide the load mask using the CheckoutComponent's Checkout succeeded signal.
